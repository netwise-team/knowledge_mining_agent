# Сценарии функционального тестирования Knowledge Mining Agent

---

## 📋 Общее описание  

Данные сценарии проверяют заявленный функционал агента в реальных бизнес-сценариях использования. Каждый сценарий описывает конкретную пользовательскую задачу и ожидаемый результат.

---

## СЦЕНАРИЙ №1: Поиск знаний в базе pm-wiki
**Название**: Поиск информации по Process Mining  

**Цель**: Проверить способность агента находить релевантные статьи в базе знаний и корректно отображать их рейтинг.  

**Предусловия**:  
- База знаний pm-wiki содержит 5 статей  
- У статей есть заполненные рейтинги

**Шаги**:
| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь пишет: "найди в вики что такое conformance checking" | Агент выполняет поиск в pm-wiki |
| 2 | | Агент находит минимум 1 статью по теме |
| 3 | | Агент показывает название статьи |
| 4 | | Агент показывает краткое содержание |
| 5 | | Агент отображает рейтинг в виде звёзд (★★★★★) |
| 6 | | Агент указывает slug статьи для ссылки |
| 7 | | Агент предлагает поставить лайк или дизлайк |
| 8 | | Агент подтверждает, какая статья использована для ответа |

**Критерии успеха**:   
✅ Найдена хотя бы одна статья  
✅ Рейтинг отображается в звёздном формате  
✅ Есть предложение обратной связи  
✅ Указан slug использованной статьи  

**Бизнес-ценность**: Проверяет, что сотрудник может быстро найти нужную информацию в корпоративной базе знаний.   

---

## СЦЕНАРИЙ №2: Оценка полезности статей  
**Название**: Обработка обратной связи (лайк/дизлайк)

**Цель**: Проверить, что система рейтингования работает и влияет на приоритетность статей в будущих ответах.

**Предусловия**:  
- В pm-wiki есть статья с рейтингом, например, 3
- Пользователь получил ответ на основе этой статьи

**Шаги**:  
| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь ставит лайк: "лайк" или "👍" | Агент определяет, какая статья использовалась в последнем ответе |
| 2 | | Агент читает текущий рейтинг статьи |
| 3 | | Агент увеличивает рейтинг на 1 |
| 4 | | Агент обновляет метаданные статьи |
| 5 | | Агент сообщает: "Спасибо! Рейтинг статьи увеличен до 4" |

**Альтернативный сценарий (дизлайк):**

| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь ставит дизлайк: "дизлайк" или "👎" | Агент определяет использованную статью |
| 2 | | Агент уменьшает рейтинг на 1 |
| 3 | | Агент сообщает: "Понял, рейтинг статьи снижен до 2. Буду реже использовать её в ответах" |

**Критерии успеха**:  
✅ Рейтинг изменился на ±1  
✅ Агент подтвердил изменение  
✅ Изменение сохранилось (проверяется повторным запросом)  

**Бизнес-ценность**: Позволяет коллективно оценивать качество знаний и автоматически улучшать ранжирование. 

---

## СЦЕНАРИЙ №3: Сортировка по рейтингу
**Название**: Приоритизация наиболее проверенных знаний

**Цель**: Проверить, что при поиске статей агент сортирует результаты по рейтингу, отдавая приоритет наиболее проверенным источникам.

**Предусловия**:  

- В базе есть минимум 3 статьи на одну тему с разными рейтингами  
- Например:  
  - Статья А: рейтинг 5  
  - Статья Б: рейтинг 3  
  - Статья В: рейтинг 1
    
**Шаги**: 
| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь пишет: "покажи все статьи по process discovery" | Агент находит все статьи по теме |
| 2 | | Агент сортирует статьи по убыванию рейтинга |
| 3 | | Первой в списке показывается статья с наивысшим рейтингом |
| 4 | | Последней — статья с наименьшим рейтингом |
| 5 | | Агент использует для ответа статью с наивысшим рейтингом |

**Критерии успеха**:  
✅ Статьи отсортированы от наивысшего рейтинга к низшему  
✅ Для ответа выбрана статья с максимальным рейтингом  
✅ В ответе указан рейтинг использованной статьи  

**Бизнес-ценность**: Гарантирует, что пользователь получает наиболее проверенную и надёжную информацию из корпоративной базы знаний.

## СЦЕНАРИЙ №6: Извлечение GitHub репозиториев
**Название**: Обнаружение программных реализаций в научных статьях
**Цель**: Проверить, что агент автоматически находит ссылки на GitHub в тексте статьи и извлекает информацию о реализации.

**Предусловия**:
- Есть статья с явной ссылкой на GitHub репозиторий  
- Репозиторий доступен и содержит README  

**Шаги**:  
| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь отправляет статью с текстом: "Our implementation is available at https://github.com/razzant/pm-toolkit" | Агент сканирует текст статьи |
| 2 | | Агент обнаруживает ссылку на GitHub |
| 3 | | Агент загружает README репозитория |
| 4 | | Агент извлекает информацию: что делает, архитектуру, алгоритмы |
| 5 | | Агент создаёт отдельную страницу в pm-wiki для репозитория |
| 6 | | Агент устанавливает ссылку от статьи к репозиторию |

**Критерии успеха**:  
✅ Ссылка на GitHub обнаружена  
✅ README успешно загружен  
✅ Создана wiki-страница репозитория  
✅ На странице есть секции: Обзор, Архитектура, Алгоритмы, Установка, Лицензия  

**Бизнес-ценность**: Автоматическое документирование программных реализаций, связанных с научными исследованиями.

## СЦЕНАРИЙ №7: Представление агента
**Название**: Приветствие и представление возможностей
**Цель**: Проверить, что агент корректно представляет себя и показывает доступный функционал.  

**Предусловия**:  
- Агент запущен и готов к работе  

**Шаги**:  
| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь пишет: "Привет! Кто ты?" | Агент представляется как специализированная версия Ouroboros |
| 2 | | Агент указывает свою специализацию — Process Mining |
| 3 | | Агент перечисляет доступные навыки (скилы) |
| 4 | | Агент проверяет подключение к базе знаний pm-wiki |
| 5 | | Агент сообщает статус подключения: "Подключено ✅" или "Не подключено ❌" |
| 6 | | Агент предлагает задать вопросы по Process Mining |

**Критерии успеха**:  
✅ В ответе есть упоминание Process Mining  
✅ Перечислены минимум 2 навыка  
✅ Показан статус подключения к базе знаний  
✅ Ответ на русском языке  

**Бизнес-ценность**: Пользователь понимает, с кем общается и какие задачи может решать агент.

## СЦЕНАРИЙ №8: Интеграция с Synthadoc
**Название**: Проверка подключения к базе знаний  
**Цель**: Проверить, что агент корректно работает с Synthadoc (базой знаний pm-wiki).  

**Предусловия**:  
- Synthadoc запущен и доступен  

**Шаги**:

| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь спрашивает: "покажи последние добавленные статьи" | Агент вызывает mcp_synthadoc__synthadoc_list_pages |
| 2 | | Агент получает список страниц |
| 3 | | Агент показывает 5–10 последних статей |
| 4 | | Для каждой статьи: название, дата добавления, рейтинг |

**Критерии успеха**:  
✅ Список страниц получен  
✅ Показаны актуальные статьи  
✅ У статей есть названия и даты  

**Бизнес-ценность**: Контроль актуальности базы знаний, возможность отслеживать новые поступления.
