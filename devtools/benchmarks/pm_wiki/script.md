# Сценарии функционального тестирования Knowledge Mining Agent

---

## 📋 Общее описание  

Данные сценарии проверяют заявленный функционал агента в реальных бизнес-сценариях использования. Каждый сценарий описывает конкретную пользовательскую задачу и ожидаемый результат.  

---

### СЦЕНАРИЙ №1: Поиск в базе знаний (pm-wiki)

**Название**: Семантический поиск по базе знаний

**Цель**: Проверить способность агента находить релевантные статьи в базе знаний pm-wiki и корректно отображать их статус.

**Соответствие навыку**: `pm-wiki`

**Предусловия**:
- База знаний pm-wiki содержит минимум 5 статей по Process Mining
- Часть статей находится в статусе "draft", часть — в "active"

**Шаги**:

| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь пишет: *"Найди в вики информацию о conformance checking"* | Агент выполняет семантический поиск по pm-wiki |
| 2 | | Агент находит минимум 1 релевантную статью |
| 3 | | Агент показывает название статьи с указанием авторов и года публикации |
| 4 | | Агент выводит краткое содержание (выдержку) статьи |
| 5 | | Агент указывает статус статьи: "active" или "draft" |
| 6 | | Агент указывает slug статьи для перехода на wiki-страницу |
| 7 | | Агент предлагает задать уточняющий вопрос по статье |

**Критерии успеха**:  
✅ Найдена хотя бы одна релевантная статья  
✅ Отображены название, авторы и год публикации  
✅ Указан статус статьи (draft/active)  
✅ Указан slug использованной статьи  

**Бизнес-ценность**: Сотрудник может быстро найти информацию в корпоративной базе знаний и понять, является ли статья финальной (active) или требует доработки (draft).

---

### СЦЕНАРИЙ №2: Изменение статуса статьи

**Название**: Перевод статьи из черновика в активную

**Цель**: Проверить, что агент может изменять статус статьи в базе знаний.

**Соответствие навыку**: `pm-wiki`

**Предусловия**:
- В pm-wiki есть статья со статусом "draft"
- Пользователь имеет права на изменение статуса

**Шаги**:

| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь пишет: *"Переведи статью 'Process Mining with Python' в active"* | Агент находит статью по названию |
| 2 | | Агент проверяет текущий статус статьи |
| 3 | | Агент изменяет статус с "draft" на "active" |
| 4 | | Агент подтверждает: *"Статья 'Process Mining with Python' переведена в статус active"* |
| 5 | Пользователь проверяет: *"Покажи статью 'Process Mining with Python'"* | Агент показывает статью со статусом "active" |

**Альтернативный сценарий (возврат в черновик)**:

| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь пишет: *"Верни статью в draft"* | Агент находит статью |
| 2 | | Агент изменяет статус с "active" на "draft" |
| 3 | | Агент подтверждает изменение статуса |

**Критерии успеха**:  
✅ Статья найдена по названию  
✅ Статус успешно изменён  
✅ Агент подтвердил изменение  
✅ Изменение сохранилось (проверяется повторным запросом)  

**Бизнес-ценность**: Управление жизненным циклом статей в базе знаний — от черновика до опубликованного материала.

---

### СЦЕНАРИЙ №3: Фильтрация по статусу

**Название**: Поиск только активных или черновых статей

**Цель**: Проверить, что агент может фильтровать результаты поиска по статусу статьи.

**Соответствие навыку**: `pm-wiki`

**Предусловия**:
- В базе есть минимум 3 статьи на тему "process discovery"
- Среди них есть как "active", так и "draft"

**Шаги**:

| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь пишет: *"Покажи все активные статьи по process discovery"* | Агент находит статьи по теме |
| 2 | | Агент фильтрует статьи со статусом "active" |
| 3 | | В результатах отображаются только статьи со статусом "active" |
| 4 | Пользователь уточняет: *"А теперь покажи черновики"* | Агент фильтрует статьи со статусом "draft" |
| 5 | | В результатах отображаются только статьи со статусом "draft" |

**Критерии успеха**:  
✅ При поиске "active" не отображаются "draft" статьи  
✅ При поиске "draft" не отображаются "active" статьи  
✅ В каждом случае количество статей соответствует реальному состоянию базы знаний  

**Бизнес-ценность**: Пользователь может работать только с проверенной информацией (active) или просматривать материалы в процессе подготовки (draft).

---

### СЦЕНАРИЙ №4: Пакетный поиск и загрузка статей

**Название**: Сбор актуальных исследований по Process Mining

**Цель**: Проверить способность агента находить и загружать научные статьи из открытых источников (ArXiv, IEEE Xplore, Google Scholar, Semantic Scholar).

**Соответствие навыку**: `pm-research`

**Предусловия**:
- Есть доступ к интернету
- Настроены API-ключи для ArXiv (и других источников по мере необходимости)

**Шаги**:

| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь пишет: *"Найди и загрузи статьи по process mining в банковской сфере за 2024 год"* | Агент выполняет поиск по указанным источникам |
| 2 | | Агент находит минимум 5 релевантных статей |
| 3 | | Агент показывает список: название, авторы, год публикации, источник |
| 4 | | Агент скачивает полные тексты статей в формате PDF |
| 5 | | Агент проверяет валидность PDF-файлов |
| 6 | | Агент проверяет загруженные статьи на дубликаты |
| 7 | | Агент сохраняет статьи в базу знаний через синтаксический анализ (synthadoc_ingest) |
| 8 | | Агент сообщает отчёт: сколько статей найдено, загружено и добавлено |

**Критерии успеха**:  
✅ Найдено минимум 5 статей  
✅ У каждой статьи есть название, авторы и год публикации  
✅ Скачаны валидные PDF-файлы (не повреждены)  
✅ Дубликаты корректно отфильтрованы  
✅ Агент предоставил отчёт о результатах загрузки  

**Бизнес-ценность**: Полная автоматизация сбора научной литературы для исследовательских и аналитических задач, исключающая ручной труд.

---

### СЦЕНАРИЙ №5: Извлечение GitHub-репозиториев

**Название**: Обнаружение программных реализаций в научных статьях

**Цель**: Проверить, что агент автоматически находит ссылки на GitHub в тексте статьи и создаёт структурированную wiki-страницу с информацией о репозитории.

**Соответствие навыку**: `pm-github-knowledge-extractor`

**Предусловия**:
- В базе знаний есть статья, содержащая явную ссылку на GitHub-репозиторий
- Репозиторий доступен и содержит README-файл

**Шаги**:

| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь загружает статью через `pm-research` или запрашивает: *"Покажи репозитории, связанные со статьей 'Process Mining with Python'"* | Агент сканирует текст статьи на наличие ссылок на GitHub |
| 2 | | Агент обнаруживает ссылку на GitHub-репозиторий |
| 3 | | Агент загружает README репозитория через curl |
| 4 | | Агент парсит информацию: архитектура, ключевые алгоритмы, зависимости, лицензия, автор |
| 5 | | Агент создаёт отдельную wiki-страницу в формате Markdown-таблицы |
| 6 | | Агент устанавливает связь между статьёй и репозиторием в базе знаний |

**Критерии успеха**:  
✅ Ссылка на GitHub обнаружена  
✅ README успешно загружен  
✅ Создана wiki-страница репозитория  
✅ На странице присутствуют секции: Обзор, Архитектура, Алгоритмы, Зависимости, Лицензия, Автор  

**Бизнес-ценность**: Автоматическое документирование программных реализаций, связанных с научными исследованиями, создание связанной экосистемы знаний "статья → код".

### СЦЕНАРИЙ №6: Представление агента

**Название**: Идентификация и проверка подключения к базе знаний

**Цель**: Проверить, что агент корректно представляет себя, свои возможности и статус подключения к базе знаний.

**Соответствие навыку**: `pm-identity`

**Предусловия**:
- Агент запущен и готов к работе
- Synthadoc подключен

**Шаги**:

| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь пишет: *"Привет! Кто ты?"* | Агент представляется как Knowledge Mining Agent |
| 2 | | Агент указывает свою специализацию — Process Mining |
| 3 | | Агент называет текущую версию (например, 6.56.4) |
| 4 | | Агент перечисляет доступные навыки (pm-identity, pm-wiki, pm-research, pm-github-knowledge-extractor) |
| 5 | | Агент проверяет подключение к базе знаний pm-wiki |
| 6 | | Агент сообщает статус: *"Подключено"* или *"Не подключено"* |
| 7 | | Агент показывает статистику базы знаний: количество статей, репозиториев, дата последнего обновления |
| 8 | | Агент предлагает задать вопросы по Process Mining |

**Критерии успеха**:  
✅ В ответе есть упоминание Process Mining  
✅ Перечислены минимум 3 навыка  
✅ Показан статус подключения к базе знаний  
✅ Отображена статистика базы знаний  
✅ Ответ на русском языке  

**Бизнес-ценность**: Новый пользователь быстро понимает, с кем общается, какие задачи может решать агент и готова ли база знаний к работе.

---

### СЦЕНАРИЙ №7: Интеграция с Synthadoc

**Название**: Проверка подключения и управления базой знаний

**Цель**: Проверить, что агент корректно работает с Synthadoc (движком базы знаний) и может выполнять операции со страницами.

**Соответствие навыку**: `pm-wiki`, `pm-identity`

**Предусловия**:
- Synthadoc запущен и доступен
- База знаний содержит минимум 10 страниц

**Шаги**:

| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь спрашивает: *"Покажи последние добавленные статьи"* | Агент вызывает MCP-клиент для работы с Synthadoc |
| 2 | | Агент получает список страниц через `synthadoc_list_pages` |
| 3 | | Агент показывает 5–10 последних добавленных статей |
| 4 | | Для каждой статьи отображается: название, дата добавления, статус (draft/active), ссылка (slug) |
| 5 | Агент также проверяет: *"Сколько всего статей в базе?"* | Агент получает общую статистику через Synthadoc |
| 6 | | Агент сообщает общее количество страниц |

**Критерии успеха**:  
✅ Список страниц получен  
✅ Показаны актуальные статьи с корректными датами  
✅ У статей есть названия и статусы  
✅ Статистика базы знаний соответствует фактическому состоянию  

**Бизнес-ценность**: Контроль актуальности базы знаний, возможность отслеживать новые поступления и мониторить наполнение корпоративной базы знаний.

---

### СЦЕНАРИЙ №8: Саммари статьи

**Название**: Генерация краткого пересказа научной статьи

**Цель**: Проверить способность агента создавать сжатое саммари статьи с выделением ключевых тезисов.

**Соответствие навыку**: `pm-wiki`

**Предусловия**:
- В базе знаний есть статья с полным текстом или аннотацией
- Агент имеет доступ к LLM для генерации саммари

**Шаги**:

| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь пишет: *"Сделай краткий пересказ статьи 'Process Mining with Python'"* | Агент находит статью в базе знаний |
| 2 | | Агент извлекает ключевые тезисы из текста |
| 3 | | Агент генерирует сжатое саммари (3–5 предложений) |
| 4 | | Агент выделяет основные результаты и выводы |
| 5 | | Агент предлагает запросить более детальный разбор |
| 6 | По запросу пользователя: *"Расскажи подробнее"* | Агент предоставляет расширенное саммари с деталями методологии |

**Критерии успеха**:  
✅ Статья найдена в базе знаний  
✅ Сгенерировано саммари на русском языке  
✅ В саммари выделены ключевые тезисы  
✅ Саммари содержит ссылку на полную статью  
✅ Агент предлагает углублённый разбор  

**Бизнес-ценность**: Быстрое ознакомление с содержанием статьи без чтения полного текста, эффективное потребление знаний.

---

### СЦЕНАРИЙ №9: Диагностика базы знаний

**Название**: Проверка состояния и выявление проблем в базе знаний

**Цель**: Проверить, что агент может провести аудит базы знаний и сообщить о выявленных проблемах.

**Соответствие навыку**: `pm-wiki`, `pm-identity`

**Предусловия**:
- База знаний содержит статьи с потенциальными проблемами (отсутствие аннотации, устаревшие данные, статус "draft" длительное время)

**Шаги**:

| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь пишет: *"Проверь состояние базы знаний"* | Агент анализирует все страницы в pm-wiki |
| 2 | | Агент проверяет наличие обязательных полей (название, авторы, год, аннотация) |
| 3 | | Агент выявляет статьи в статусе "draft" старше 30 дней |
| 4 | | Агент проверяет дату последнего обновления статей |
| 5 | | Агент формирует отчёт: общая статистика, список проблем, рекомендации |
| 6 | | Агент предлагает исправить выявленные нарушения |

**Критерии успеха**:  
✅ Агент провёл полный аудит базы знаний  
✅ Выявлены статьи с проблемами  
✅ Предоставлена статистика (всего статей, active, draft, устаревших)  
✅ Даны рекомендации по улучшению качества базы знаний  

**Бизнес-ценность**: Поддержание базы знаний в актуальном и качественном состоянии, своевременное выявление проблемных областей.

---

### СЦЕНАРИЙ №10: Автоматическое обновление базы знаний

**Название**: Мониторинг новых публикаций и пополнение базы знаний

**Цель**: Проверить, что агент в автономном режиме может отслеживать новые публикации и добавлять их в базу знаний.

**Соответствие навыку**: `pm-research`, `pm-wiki`

**Предусловия**:
- Настроено расписание в интерфейсе агента
- Указаны ключевые слова для поиска
- Настроен канал для уведомлений (Telegram)

**Шаги**:

| № | Действие пользователя | Ожидаемый результат агента |
|---|----------------------|---------------------------|
| 1 | Пользователь настраивает расписание: *"Каждую неделю ищи статьи по process mining"* | Агент сохраняет настройки расписания |
| 2 | | Агент запускается по расписанию в фоновом режиме |
| 3 | | Агент выполняет поиск новых статей по заданным ключевым словам |
| 4 | | Агент проверяет, есть ли эти статьи уже в базе знаний |
| 5 | | Агент загружает новые статьи и сохраняет их со статусом "draft" |
| 6 | | Агент отправляет отчёт с результатами (в Telegram) |
| 7 | Пользователь получает отчёт и может опубликовать статьи (перевести в "active") или отредактировать их |

**Критерии успеха**:  
✅ Настройки расписания сохранены корректно  
✅ Агент запустился по расписанию  
✅ Найдены новые статьи (если они есть)  
✅ Новые статьи добавлены в базу знаний со статусом "draft"  
✅ Отправлен отчёт с результатами  
✅ Статьи не дублируются (повторный запуск не добавляет копии)  

**Бизнес-ценность**: База знаний всегда пополняется свежими материалами без ручного вмешательства, а ответственный сотрудник получает уведомление и может проверить качество новых статей перед публикацией.
